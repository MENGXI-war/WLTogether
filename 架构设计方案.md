# 网络媒体服务 — 架构设计方案

## 用户决策
- **媒体存储**: 客户端本地存储，服务器不存媒体文件
- **文件分发**: P2P 传输（WebTorrent），绕过服务器带宽瓶颈
- **服务器角色**: 纯协调中枢——认证、群组、授时同步、聊天、P2P信令
- **用户体系**: 双模式 — 注册用户（完整功能）+ 设备用户（P2P 自组网，仅签发证书）

---

## 一、核心理念：服务器不碰媒体文件

```
┌──────────────────────────────────────────────────────────┐
│                     服务器 (纯协调)                        │
│   ┌──────────┐  ┌──────────┐  ┌──────────────────┐      │
│   │ 用户认证  │  │ 群组管理  │  │ 播放授时同步      │      │
│   │ JWT      │  │ 成员/角色 │  │ 时钟同步+指令广播  │      │
│   └──────────┘  └──────────┘  └──────────────────┘      │
│   ┌──────────┐  ┌──────────┐  ┌──────────────────┐      │
│   │ 聊天消息  │  │ P2P信令  │  │ 会话元数据        │      │
│   │ 持久化    │  │ 中继     │  │ (hash/磁力链接)    │      │
│   └──────────┘  └──────────┘  └──────────────────┘      │
│                                                          │
│            ★ 服务器不存储、不传输媒体文件 ★                 │
└──────────────┬───────────────────────────────┬───────────┘
               │ WebSocket + REST              │
    ┌──────────▼──────────┐          ┌────────▼───────────┐
    │   客户端 A (发起者)   │          │  客户端 B/C (加入者)  │
    │  本地媒体文件        │          │  本地媒体文件         │
    │  ↓ 创建种子          │          │  ↓ WebTorrent 下载   │
    │  通过 P2P 分发 ──────┼────P2P──►│  边下边播 / 本地已有   │
    │  按服务器授时播放     │          │  按服务器授时播放      │
    └─────────────────────┘          └────────────────────┘
```

**关键数据流：**
- 媒体文件：**永远只在客户端之间P2P传输**，不走服务器
- 服务器只传：JSON 元数据、播放时间戳、聊天消息、信令数据（几KB级别）
- 如果客户端本地已有相同文件（hash 匹配），直接跳过下载，立即加入同步播放

## 三、同步播放完整流程

```

   │                               │                               │

   │──2. 计算 SHA256 hash          │                               │
   │──3. 创建种子(.torrent+magnet)  │                               │
   │──4. 发起会话{hash, magnet,     │                               │
   │     transferMode, 文件名,时长}─►│                               │
   │                               │──5. 广播会话─────────────────►│
   │                               │   {含magnet + .torrent文件}    │
   │                               │                               │
   │                               │   ┌─6. 获取文件 (5选1) ──────┐  │
   │                               │   │  A) P2P直连 (WebTorrent) │  │
   │                               │   │  B) 种子下载 (私有tracker)│  │

   │                               │   │  本地已有 → hash匹配跳过  │  │
   │   ◄══════ 文件数据 ─────────────►│   └────────────────────────┘  │
   │                               │                               │
   │──7. 点击播放──────────────────►│                               │

   │   [本地播放器从 position=0 播放] │   [本地播放器从 position=0 播放]  │
   │                               │                               │

|------|--------|--------|--------|
| VideoSessionView | 全屏播放区 | 聊天 + 播放列表 | 进度条、字幕、分辨率 |

## 四、P2P 传输方案：WebTorrent

### 4.1 WebTorrent

Web 端通过 WebTorrent 在浏览器内实现 BitTorrent over WebRTC，支持边下边播和多人同时传输。Android 端使用 libtorrent。

WebTorrent 通过 WebRTC data channel 传输数据，需要信令服务器交换 SDP 连接信息。服务端通过 WebSocket 中继信令，不传输文件数据。

```
A的浏览器 ──WebSocket──► 服务器 ◄──WebSocket── B的浏览器
    │                                                    │
    └──────────── P2P 数据 (WebRTC Data Channel) ────────┘
```

### 4.2 Hash 去重机制
每个文件在客户端计算 SHA256（只算前1MB+后1MB+文件大小，快速去重），相同 hash 的文件视为同一文件。用户 B 本地已有该文件时，无需下载直接参与同步播放。

### 4.3 种子做种（私有 tracker）

除 WebRTC data channel 直连外，客户端也可通过传统 BitTorrent 种子方式分发文件。发起者创建 .torrent 文件时指定服务器内置的私有 tracker，接收方通过磁力链接或 .torrent 文件加入下载。

```
发起者 A                          私有 tracker (服务器)              加入者 B
  │                                     │                                │
  │──1. 创建种子(.torrent)              │                                │
  │    指定私有 tracker URL              │                                │
  │    生成 magnet URI                  │                                │
  │                                     │                                │
  │──2. 广播 magnet URI─────────────►  │──3. 广播────────────────────►│
  │    (通过 WebSocket 会话消息)         │   {magnetUri, fileName, ...}    │
  │                                     │                                │
  │──4. 开始做种─────────────────────►  │◄──5. 请求 peer 列表─────────│
  │    (向 tracker 注册，上报 IP:port)    │     tracker 返回 A 的地址       │
  │                                     │                                │
  │◄══════ 6. P2P 连接 + 数据交换 ════════════════════════════════►  │
  │         MSE/PE 加密握手              │                                │
  │         支持边下边播 (顺序下载)        │                                │
```

| 方面 | 设计 |
|------|------|
| Tracker 类型 | HTTP tracker，服务器内置，`/api/tracker/announce` |
| Tracker 可见性 | 仅群组成员可访问，JWT 鉴权 |
| 加密 | 强制 MSE/PE 协议加密（BitTorrent 协议层） |
| DHT / PEX | **关闭**（只走私有 tracker，不泄露到公网） |
| 做种范围 | 仅群内在线成员，全员下载完成后 tracker 停止返回 peer |
| 安全 | 非群组成员即使拿到 magnet URI 也无法连接 tracker |

**与 P2P 直连的区别：**

| | WebRTC 直连 (4.1) | 私有 tracker 做种 (4.3) |
|------|------|------|
| 传输协议 | WebRTC Data Channel (SRTP) | TCP/uTP (BitTorrent) |
| NAT 穿透 | STUN/TURN | uTP 自带 NAT 穿越，弱于 WebRTC |
| 加密 | DTLS-SRTP（WebRTC 强制） | MSE/PE（BitTorrent 协议层） |
| 适用场景 | Web 端、浏览器环境 | Android 端、大文件、做种分担上行 |
| 边下边播 | 支持（数据块顺序优先） | 支持（libtorrent 优先级队列） |

### 4.4 备选方案：离线打包分发

用户可以将媒体文件打包成加密的标准化包，自行通过任意渠道（网盘/QQ/微信/U盘/局域网等）传给其他群成员。安全认证通过群组内密钥分发保证，只有目标群组成员能解包导入。

**核心思路：** 客户端只管**打包**和**导入**，传输渠道由用户自行解决。安全由三层可选防护保证——群组绑定、内容加密、数字签名。所有安全相关密钥通过服务器 WebSocket（WSS 加密通道）在群组内分发。

```
发起者 A                                    服务器                        加入者 B
  │                                           │                              │
  │──1. 打包设置─────────────────────────────►│                              │
  │    选择媒体文件 + 安全选项:                 │                              │
  │    [✓] 群组绑定    (默认开启)               │                              │
  │    [✓] 内容加密    (默认开启)               │                              │
  │    [ ] 数字签名    (可选)                   │                              │
  │                                           │                              │
  │──2. 客户端自动:                            │                              │
  │    生成随机 AES-256 密钥                    │                              │
  │    加密媒体文件 + manifest                  │                              │
  │    生成 SHA256 校验值                       │                              │
  │    打包为 .wlp 文件 ──►                     │                              │
  │                                           │                              │
  │──3. 密钥上传─────────────────────►         │                              │
  │    {groupId, packageHash,                   │                              │
  │     aesKey(加密后的), signature?}            │──4. 群内广播密钥─────────►  │
  │                                           │   (仅群成员可见)              │
  │                                           │                              │
  │──5. 通过任意渠道传包 ── ── ── ── ── ── ── ── ── ── ── ── ──►         │
  │   (网盘/QQ/微信/U盘/飞书/邮件…)              │                              │
  │                                           │                         ┌─6. 导入包──┐
  │                                           │                         │ 选择 .wlp  │
  │                                           │                         │ 自动从群聊  │
  │                                           │                         │ 获取密钥    │
  │                                           │                         │ 解密→验签  │
  │                                           │                         │ →验hash    │
  │                                           │                         │ →解压导入  │
  │                                           │                         └────────────┘
```

**.wlp 包结构：**

```
示例文件_1080p.wlp  (本质是 zip，内容可能加密)
│
├── encrypted/                          # 若启用加密：AES-256-GCM 加密区
│   ├── 示例文件_1080p.mkv.enc          # 加密的媒体文件
│   └── manifest.json.enc              # 加密的清单
│
├── manifest.sig                        # 若启用签名：manifest 的数字签名
├── package.json                        # 包头（明文，供客户端识别）
└── public_info.json                    # 公开信息（不加密，预览用）
```

**package.json（包头，始终明文）：**
```json
{
  "format": "wltogether-package",
  "version": 1,
  "appId": "com.wltogether.client",
  "groupId": "abc123",
  "groupName": "我的群组",
  "packageHash": "sha256:def456...",
  "security": {
    "encrypted": true,
    "signed": true,
    "groupBound": true
  },
  "exporter": {
    "userId": "u_001",
    "nickname": "张三"
  },
  "createdAt": "2026-05-31T12:00:00Z"
}
```

**public_info.json（加密前文件信息，供接收者预览）：**
```json
{
  "fileName": "示例文件_1080p.mkv",
  "fileSize": 5368709120,
  "duration": 10140,
  "mediaType": "VIDEO",
  "videoCodec": "h264",
  "width": 1920,
  "height": 1080
}
```

### 4.4.1 三层安全防护

用户打包时可自由勾选，每层独立开关：

#### 第一层：群组绑定（默认 ✅）

| 项目 | 说明 |
|------|------|
| 原理 | package.json 内嵌 `groupId`，导入时客户端校验「当前用户是否在该群组内」 |
| 防什么 | 非群组成员拿到包文件无法导入；包文件被转发到其他群无法使用 |
| 密钥分发 | 不需要额外密钥，服务器直接校验群成员身份 |
| 破解难度 | 低（修改 package.json 中的 groupId 即可绕过，仅防君子） |

#### 第二层：内容加密（默认 ✅）

| 项目 | 说明 |
|------|------|
| 算法 | AES-256-GCM（带认证的对称加密，同时保证机密性和完整性） |
| 密钥生成 | 打包时客户端随机生成 256-bit 密钥 |
| 密钥分发 | 密钥通过 WSS 加密通道上传服务器 → 服务器仅转发给**该群组当前在线成员**，不持久化存储密钥 |
| 离线成员 | 下次上线后，客户端自动拉取未消费的密钥（服务器暂存，24h 后自动清除） |
| 防什么 | 网盘/传输渠道被第三方看到内容；文件在传输途中被篡改（GCM 自带完整性校验）；运营商/云盘扫描文件内容 |
| 破解难度 | 高（AES-256 暴力破解不可行） |

**密钥转发协议（服务器不存密钥明文）：**
```
打包者 → 服务器                      服务器 → 群成员
  {                              {
    type: "package.key",           type: "package.key",
    groupId: "abc123",             packageHash: "sha256:def456...",
    packageHash: "sha256:def456",  exporterId: "u_001",
    key: "<AES-256 key>",         key: "<AES-256 key>",
    expiresIn: 86400              expiresAt: "..."
  }                              }
```
服务器仅在内存中暂存密钥（24 小时 TTL），到期自动清除。密钥不在数据库中持久化，也不写入日志。

#### 第三层：数字签名（可选 ⬜）

| 项目 | 说明 |
|------|------|
| 算法 | Ed25519（椭圆曲线签名，快、短、安全） |
| 密钥对 | 客户端首次使用时自动生成 Ed25519 密钥对，公钥上传到服务器用户资料 |
| 签名内容 | 对 `manifest.json` 的 SHA256 做签名，生成 `manifest.sig` |
| 验签 | 导入时客户端从服务器获取导出者的公钥，验证签名 |
| 防什么 | **身份冒充**——有人伪造张三的包发给群友，签名不匹配则拒绝导入；**内容篡改后重新打包**——改了 manifest 重签名不可能（没有私钥） |
| 破解难度 | 极高（Ed25519 不可伪造） |

### 4.4.2 导入时安全校验流程

```
收到 .wlp 文件
      │
      ▼
┌─ 1. 解析 package.json ──────────────────────────────┐
│   ├─ 校验 format = "wltogether-package"              │
│   ├─ 校验 appId = "com.wltogether.client"            │
│   └─ 读取 security 开关                              │
└─────────────────────────────────────────────────────┘
      │
      ▼
┌─ 2. 群组绑定检查 (若 groupBound = true) ─────────────┐
│   当前用户是否在 package.json 指定的 groupId 中？      │
│   ✗ 否 → "此包绑定到群组「xxx」，你不在该群中，无法导入"  │
└─────────────────────────────────────────────────────┘
      │
      ▼
┌─ 3. 获取解密密钥 (若 encrypted = true) ───────────────┐
│   客户端自动通过 WSS 向服务器请求 packageHash 对应的密钥  │
│   ✗ 密钥已过期(>24h) → "密钥已过期，请联系打包者重新分享" │
│   ✗ 未找到 → "密钥不可用"                              │
└─────────────────────────────────────────────────────┘
      │
      ▼
┌─ 4. 解密 + 完整性校验 (若 encrypted = true) ──────────┐
│   AES-256-GCM 解密 → GCM tag 自动校验完整性            │
│   ✗ tag 不匹配 → "文件已损坏或被篡改，拒绝导入"          │
└─────────────────────────────────────────────────────┘
      │
      ▼
┌─ 5. 签名验证 (若 signed = true) ────────────────────┐
│   客户端从服务器获取导出者公钥 → 对 manifest hash 验签   │
│   ✗ 签名无效 → "此包的签名无效，可能已被篡改或非本人打包" │
│   ✗ 签名者 ≠ package.json.exporter.userId → "签名者不匹配"│
└─────────────────────────────────────────────────────┘
      │
      ▼
┌─ 6. 文件安全扫描 ────────────────────────────────────┐
│   ├─ 校验解压后的文件扩展名在白名单内                    │
│   ├─ 拒绝 .exe .sh .bat .cmd .ps1 .app .dmg .apk 等  │
│   ├─ 拒绝路径穿越 (../, 绝对路径)                       │
│   └─ 包大小 ≤ 用户设置的上限 (默认 10GB)                 │
└─────────────────────────────────────────────────────┘
      │
      ▼
┌─ 7. Hash 校验 ───────────────────────────────────────┐
│   SHA256(解压后的媒体文件) == manifest 中的 sha256？    │
│   ✗ 不匹配 → "文件 hash 不一致，拒绝导入"               │
└─────────────────────────────────────────────────────┘
      │
      ▼
   导入成功 → 注册到本地媒体库 → P2P 共享可用
```

### 4.4.3 私钥文件导出（.wlk — 防 24h 过期）

WSS 在线密钥分发有 24 小时 TTL 限制。为避免接收者错过窗口期，打包者可选择**导出 .wlk 私钥文件**，通过任意渠道（同 .wlp 包一起）传给接收者。接收者用 .wlk 文件 + 密码即可离线解密，不依赖服务器。

**.wlk 文件格式：**

```
示例文件.wlk  (JSON 明文头 + AES-256-GCM 加密体)
│
├── header (明文 JSON):
│   {
│     "format": "wltogether-key",
│     "version": 1,
│     "packageHash": "sha256:def456...",
│     "exporterId": "u_001",
│     "exporterNickname": "张三",
│     "createdAt": "2026-05-31T12:05:00Z",
│     "salt": "<Base64随机盐>",
│     "kdf": "PBKDF2-HMAC-SHA256",
│     "kdfIterations": 100000
│   }
│
└── encrypted (AES-256-GCM):
    {
      "key": "<AES-256-GCM密钥>",        // 解密 .wlp 包的实际密钥
      "packageHash": "sha256:def456...",  // 二次校验，防止密钥被错配到别的包
      "expiresIn": 86400                  // 原服务器 TTL，仅作提示
    }
```

**导出流程：**
```
打包者                              接收者
  │                                   │
  │──1. 打包完成后选择"导出私钥"──►    │
  │                                   │
  │──2. 输入导出密码（≥8字符）          │
  │    客户端自动：                     │
  │    生成随机盐 → PBKDF2 派生       │
  │    用派生密钥 AES-256-GCM 加密     │
  │    → 生成 .wlk 文件                │
  │                                   │
  │──3. 通过任意渠道传 .wlk ── ── ──►│
  │   (网盘/QQ/微信/U盘/邮件...)       │
  │                                   │
  │                                   │──4. 导入时选择 .wlk 文件
  │                                   │    输入导出密码
  │                                   │    客户端自动：salt+KDF 派生→解密
  │                                   │    → 获得 AES 密钥 → 解密 .wlp 包
```

**校验规则：**
| 步骤 | 校验项 | 失败处理 |
|------|--------|----------|
| 1 | `format = "wltogether-key"` | "无效的私钥文件" |
| 2 | `packageHash` 与待解密的 .wlp 包一致 | "私钥文件与包不匹配" |
| 3 | 密码派生 → AES-GCM 解密 → tag 校验 | "密码错误或文件损坏" |
| 4 | 解密后 payload.packageHash == header.packageHash | "私钥文件完整性校验失败" |

**安全说明：**
- .wlk 文件依赖密码强度保护，不含密码则无法解密
- 密码不存储在文件内，不经过服务器
- 建议密码通过独立渠道（电话/当面）告知，不与 .wlp / .wlk 同渠道传输
- 服务器全程不接触 .wlk 文件，不感知导出密码

### 4.5 可选方案：服务器中转传输（需高带宽服务器）

如果未来升级到 200Mbps 带宽服务器，可以开启服务器中转模式——发起者上传文件到服务器，服务器向下分发给群组成员。这是最传统也最简单可靠的方案。

> **注意：此方案在当前 3Mbps 服务器上不可用。** 管理员在后台开启后才会生效；未开启时客户端不可见此选项。

```
发起者 A                        服务器                          加入者 B/C
  │                               │                               │
  │──1. 发起会话 + 上传文件──►     │                               │
  │    POST multipart              │                               │
  │    显示上传进度                │──2. 存储到临时目录             │
  │                               │    计算 SHA256                │
  │                               │──3. 广播会话─────────────────►│
  │                               │    {fileName, size, hash,      │
  │                               │     downloadUrl}               │
  │                               │                               │
  │   [上传完成，开始播放]          │                               │
  │                               │   ┌─4. 下载文件──────────┐    │
  │                               │   │ GET /api/files/{id}   │    │
  │                               │   │ 支持 HTTP Range       │    │
  │                               │   │ 支持断点续传           │    │
  │                               │   │ 可边下边播 (Range请求) │    │
  │                               │   └──────────────────────┘    │
  │                               │                               │
  │──5. 播放控制──────────────►   │──6. 广播同步指令──────────►  │
  │                               │                               │
  │                               │──7. 全员下载完成              │
  │                               │    自动清理临时文件            │
```

**服务器资源估算（200Mbps = 25MB/s）：**

| 场景 | 计算 | 结论 |
|------|------|------|
| 1 人上传 + 3 人同时下载 | 上行撑满 25MB/s，每人分 ~8MB/s | ✅ 4K 原盘都够 |
| 1 人上传 + 10 人同时下载 | 每人 ~2.5MB/s | ✅ 1080p 高码率够用 |
| 5GB 文件上传耗时 | 5000MB / 25MB/s ≈ 200 秒 | 约 3 分钟 |
| 存储占用（单会话） | 一个媒体文件 | 5~20GB 临时占用 |

**生命周期管理：**

| 阶段 | 行为 |
|------|------|
| 上传中 | 写入临时目录 `/tmp/wltogether/uploads/{sessionId}/` |
| 会话进行中 | 文件保留，支持 HTTP Range 随机读取 |
| 会话结束 | 标记为待清理，保留 24 小时后物理删除 |
| 24 小时后 | 定时任务（每小时执行）自动物理删除 |
| 上传中断/放弃 | 30 分钟无活动自动清理残留文件 |

**安全控制：**

| 措施 | 说明 |
|------|------|
| 下载鉴权 | 校验 JWT + 群组成员身份，非群成员拒绝下载 |
| 单文件上限 | 默认 20GB，管理员可在后台调整 |
| 并发下载限制 | 单文件最多 20 个并发连接 |
| 下载限速 | 单连接限速 10MB/s，防止占满带宽 |
| 文件类型白名单 | 上传时校验 MIME + 扩展名，仅允许视频/音频 |
| 临时目录隔离 | 上传/下载目录与系统隔离，独立分区 |

**开启方式：** 管理员在后台「系统设置」中开启，客户端通过 `GET /api/server/capabilities` 查询可用性。

### 4.6 备选方案：线下局域网传输

同一局域网下的设备通过 HTTP/TCP 直接传输文件，不经过互联网，不消耗服务器带宽。Android 端首发，后续桌面端和鸿蒙端同步支持。

**技术选型：**

| 技术 | 用途 | 说明 |
|------|------|------|
| Android NSD (Network Service Discovery) | 服务发现 | 局域网内自动广播/发现其他 WLTogether 客户端 |
| NanoHTTPD / 内置轻量 HTTP Server | 文件传输 | 发送方启动临时 HTTP 服务，接收方 GET 下载 |
| mDNS | 跨平台发现 | 桌面端/鸿蒙端加入时统一使用，无需改方案 |

```
发送方 A                                 接收方 B
  │                                         │
  │──1. 发起局域网共享──►                    │
  │    选中本地文件                           │
  │    客户端启动临时 HTTP 服务器               │
  │    通过 NSD 注册服务:                      │
  │      name: "WLTogether-张三"              │
  │      type: "_wltransfer._tcp"             │
  │      port: 随机端口                        │
  │                                         │
  │                              ◄──2. NSD 发现──│
  │                                 B 的客户端扫描局域网
  │                                 发现 "WLTogether-张三"
  │                                 显示在「附近设备」列表
  │                                         │
  │──3. 广播会话(WebSocket)────────►──────►  │
  │    {transferMode: "lan",                 │
  │     lanHost: "192.168.1.5",              │
  │     lanPort: 54321,                      │
  │     fileHash: "...",                      │
  │     fileName: "..."}                      │
  │                                         │
  │                              ◄──4. GET 下载──│
  │   [HTTP 200 + 文件流] ─────────►          │
  │   支持断点续传 (Range)                     │
  │   显示进度 + 速度                          │
  │                                         │
  │──5. 下载完成，双方就位──►                 │
  │    等待同步播放                            │
```

**关键设计：**

| 方面 | 设计 |
|------|------|
| 服务发现 | Android NSD 注册 + 监听，进入群组页面自动扫描 |
| 传输协议 | HTTP over TCP，支持 Range 断点续传 |
| 安全性 | LAN 内默认不加密（隔墙攻击不现实），可选 TLS |
| 群组信息隔离 | NSD 注册名仅包含用户昵称，不泄露群组信息；实际 IP + 端口通过 WSS 加密通道传给群成员 |
| 自动关闭 | 最后一个接收方下载完成 / 发送方离开群组 / 切后台 5 分钟后自动停止 HTTP 服务 |
| 并发 | 单 HTTP 服务支持 3~5 个接收方同时下载 |
| Wi-Fi 锁定 | 传输期间锁定 WiFi，防止系统切到移动数据 |
| 手动连接 | 桌面端/Web 端可通过输入「192.168.1.5:54321」手动连接下载 |

### 数据库双兼容策略

```
application-mysql.yml          application-sqlite.yml
        │                              │
        └──────────┬───────────────────┘
                   │
            Spring Profile
         mysql / sqlite 二选一
                   │
         ┌─────────▼─────────┐
         │  Spring Data JPA   │  ← 同一套 Entity，不写原生 SQL
         │  + Flyway 迁移      │  ← 每种方言各一套迁移脚本
         └────────────────────┘
```

**兼容原则：**
1. JPA Entity 统一，不针对特定数据库写原生 SQL
2. Flyway 迁移脚本分目录：`db/migration/mysql/` 和 `db/migration/sqlite/`
3. 自增主键用 `IDENTITY` 策略（两者都支持）
4. 时间类型用 `java.time.Instant` 映射，JPA 自动适配
5. TEXT 类型两者通用，无需特殊处理
6. `application.yml` 中 `spring.datasource.url` 按 profile 切换

**适用场景：**
| 场景 | 推荐数据库 | 原因 |
|------|-----------|------|
| 本地开发 | SQLite | 零安装，一个文件即数据库 |
| 轻量部署（2GB 内存服务器） | SQLite | 免进程，省 200MB+ 内存 |
| 高并发/多实例部署 | MySQL | 并发写入性能更好 |
| 需要外部管理工具 | MySQL | 生态工具丰富 |

## 七、用户体系（双模式）

服务端同时支持两种完全不同的用户模式，两种模式互相隔离——注册用户看不到设备用户，设备用户看不到注册用户，不可混用。

### 7.1 模式对比

| | **注册用户（完整模式）** | **设备用户（P2P 模式）** |
|------|------|------|
| 身份凭证 | email + username + 密码 → JWT | 设备指纹 → Ed25519 密钥对 → 服务端签发的证书 |
| 服务器角色 | 认证 + 群组 + 会话协调 + 聊天持久化 + 信令 + 授时 | **仅签发证书 + 临时组网中介**，组网后断联 |
| 群组 | 服务端持久化，成员角色管理 | 输入相同字符串 → 生成 groupId，P2P 自发组网，服务端不知群组存在 |
| 会话协调 | 服务端转发播放指令 + 校验模式权限 | P2P 自行协调（组内选举 host，自行转发） |
| 时钟同步 | 客户端 ↔ 服务端 NTP 采样 | 组内自行 NTP 选举（最小 RTT 者为准）或走公网 NTP |
| 聊天 | 服务端持久化 + 广播 | P2P 直发，端到端加密（证书公钥），不持久化 |
| 文件传输 | 5 种方式（P2P/种子/离线包/LAN/服务器中转） | P2P 直连 + LAN + 离线包，无服务器中转、无私有 tracker |
| 离线包安全 | 服务器密钥中转 + 群组绑定（三层） | P2P E2E 密钥分发 + groupId 绑定（三层，同注册用户），.wlk 私钥文件备用 |
| 数据留存 | 消息/会话记录/歌单全部持久化 | **零服务端留存**——服务器不知道谁跟谁组了群、播了什么、聊了什么 |

### 7.2 设备用户：证书签发

```
设备首次启动
      │
      ▼
客户端本地生成:
  deviceId  = SHA256(CPU型号 + OS版本 + 设备序列号 + 随机种子)
              // 卸载重装后 deviceId 不变（硬件信息不变）
  keyPair   = Ed25519 密钥对
  publicKey = keyPair.public (Base64)
      │
      ▼
POST /api/device/certificate
  body: { deviceId, publicKey }
  // 无需认证
      │
      ▼
服务端:
  校验 deviceId 格式（64 位 hex）
  校验 publicKey 格式
  签发证书:
    {
      deviceId,
      publicKey,
      issuedAt:  "2026-06-01T12:00:00Z",
      expiresAt: "2026-06-08T12:00:00Z",   // 7 天
      signature: Ed25519.sign(serverPrivKey, 上述字段拼接)
    }
  返回证书 → 立即遗忘（不写 users 表，不记日志）
      │
      ▼
客户端本地保存证书 → 7 天内无需联网即可凭证书参与 P2P 组网
```

| 配置项 | 值 | 说明 |
|--------|-----|------|
| 证书有效期 | 7 天 | 过期后需联网重签 |
| 服务端存储 | 无 | 不写库、不记日志，签发即忘 |
| 限流 | 同一 IP 每小时 10 次 | 防止恶意签发刷证书 |
| 私钥 | 服务端启动时生成 Ed25519 密钥对，仅内存 | 重启后密钥变更，所有旧证书失效 |

### 7.3 设备用户：组网流程

```
设备 A                 服务器（临时中介）              设备 B
  │                         │                           │
  │──1. 约定组串─────────────│                           │
  │   (A和B私下沟通，比如      │                           │
  │    "movie-night-2026")    │                           │
  │                         │                           │
  │──2. 计算 groupId─────────│                           │
  │   SHA256(组串)            │                           │
  │                         │                           │
  │──3. POST /api/device/rendezvous/offer─────────────►│
  │   {groupId, cert,                          │
  │    peerInfo: {ip:"1.2.3.4", port:12345}}   │
  │                         │                           │
  │                         │──4. 存储: 内存 Map         │
  │                         │   {groupId → [peers]}     │
  │                         │   30s TTL，定时清理        │
  │                         │                           │
  │                         │◄──5. POST /api/device/rendezvous/query────│
  │                         │   {groupId, cert}         │
  │                         │                           │
  │                         │──6. 验证双方证书签名有效─────│
  │                         │    返回现有 peer 列表       │
  │                         │    将该设备加入 peer 列表   │
  │                         │                           │
  │◄──7. 返回 peer 列表──────│──────返回 peer 列表─────►│
  │                         │                           │
  │                         │                           │
  │──8. 组网完成，服务端断联──────服务端断联────────────│
  │   后续所有通信 P2P 直连     清理该 groupId 缓存       │
  │                         │                           │
  ├──────────── 9. P2P 直连通信 ───────────────────────►│
  │   时钟选举 / 播放同步 / 聊天(E2E) / 文件传输          │
  │   服务器完全不可见                                  │
```

**组串规则：**
- 组串为任意可打印字符串，长度 6-128 字符
- 建议格式：`约定词-日期-编号`（如 `movienight-0601-001`），降低碰撞概率
- `groupId = SHA256(组串)`，服务端以此为 key 匹配 peer
- 组串本身不应在群组外传播，知晓组串即等于可加入群组

**临时中介行为：**

| 行为 | 说明 |
|------|------|
| offer 请求 | 提交 cert + peerInfo，服务端验签 cert，有效则加入 groupId 对应的 peer 列表 |
| query 请求 | 提交 cert，服务端验签，返回该 groupId 下所有 peer 列表（不含自身） |
| TTL | peer 信息 30s 过期，需定时 refresh（客户端 15s 刷新一次） |
| 断开 | 服务端发现 30s 未刷新即清除，不通知其他 peer |
| groupId 隔离 | 不同组串对应的 groupId 完全隔离，互不可见 |

### 7.4 设备用户：P2P 自运行协议

组网建立后，所有功能 P2P 自行完成，服务端不参与：

#### 7.4.1 时钟选举

```
组内各设备每 5 秒广播 ping{timestamp}
收集所有 peer RTT → RTT 最小的设备当选 time leader
其余设备计算与 leader 的 offset，用于同步播放
若 leader 掉线（3 次 ping 超时）→ 自动重新选举
若所有设备均无公网 NTP → 以 leader 本地时钟为准
```

#### 7.4.2 播放同步

```
沿用与注册用户相同的协议语义，但由 time leader 代替服务器：
  host 发出 playback.play/pause/seek   → 广播给组内所有 peer
  time leader 发出 sync.pong           → 代替服务器的授时角色
  三种播放模式（HOST_ONLY/FREE/SHARED_CONTROL）保持一致
```

#### 7.4.3 聊天（端到端加密）

```
发送方:
  消息体 {content, timestamp}
  → 用接收方证书公钥（Ed25519→X25519 转换）做 ECDH 派生对称密钥
  → AES-256-GCM 加密
  → P2P 发送密文

接收方:
  → 用自己的私钥 ECDH 派生对称密钥
  → AES-256-GCM 解密
  → 验证完整性

广播消息: 对每个 peer 分别加密，逐个发送
消息不持久化: 接收后仅内存展示，关闭即消失
```

| 加密层 | 算法 | 说明 |
|--------|------|------|
| 密钥交换 | ECDH (X25519) | Ed25519 公钥可转换为 X25519 |
| 对称加密 | AES-256-GCM | 每次会话独立派生 |
| 前向安全性 | ✅ | 每次聊天会话重新 ECDH，历史消息不可追溯 |

#### 7.4.4 文件传输

```
支持:
  - P2P 直连（WebTorrent / libtorrent）
  - 局域网传输（NSD + NanoHTTPD / mDNS）
  - 离线包 .wlp + .wlk（三层安全，见 7.4.5）

不支持:
  - 私有 tracker 做种（无私有的 tracker 服务器）
  - 服务器中转（无账号，无法鉴权）
```

#### 7.4.5 离线包（设备用户适配）

设备用户的离线包与注册用户共用同一套 .wlp 格式和三层安全模型，但密钥分发和校验链路完全走 P2P：

```
打包者 A                                              接收者 B
  │                                                     │
  │──1. 选择文件，勾选安全选项：                          │
  │    [✓] groupId 绑定    (绑定到 SHA256(组串))          │
  │    [✓] 内容加密         (AES-256-GCM)                 │
  │    [ ] 数字签名         (可选，用设备证书私钥)          │
  │                                                     │
  │──2. 客户端生成 AES-256 密钥 → 加密媒体文件             │
  │    package.json 内嵌 groupId（非服务器群组ID）          │
  │    生成 .wlp 文件                                     │
  │                                                     │
  │──3. 密钥分发（二选一）：                               │
  │    A) P2P E2E 通道（默认）:                            │
  │       用接收方证书公钥加密 AES 密钥                      │
  │       → 通过 E2E 聊天通道发送                          │
  │       → 无需服务器参与                                 │
  │    B) .wlk 私钥文件（备用）:                            │
  │       导出 .wlk（PBKDF2 + 密码加密）                   │
  │       → 通过任意渠道传输                               │
  │                                                     │
  │──4. 通过任意渠道传 .wlp ── ── ── ── ── ── ──►      │
  │                                                     │
  │                                                     │──5. 导入校验（7 步）：
  │                                                     │    ① 解析 package.json
  │                                                     │    ② groupId 绑定检查：
  │                                                     │       package.groupId == SHA256(本组组串)?
  │                                                     │       （证明包是给这个 P2P 群组的）
  │                                                     │    ③ 获取 AES 密钥：
  │                                                     │       E2E 通道接收 或 .wlk + 密码
  │                                                     │    ④ AES-256-GCM 解密 + 完整性校验
  │                                                     │    ⑤ Ed25519 验签（若签了）：
  │                                                     │       公钥取自打包者的设备证书
  │                                                     │       （组网时已交换，无需服务器查询）
  │                                                     │    ⑥ 文件安全扫描
  │                                                     │    ⑦ SHA256 校验
  │                                                     │       → 注册到本地媒体库
```

**与注册用户离线包的区别：**

| 环节 | 注册用户 | 设备用户 |
|------|----------|----------|
| 密钥分发 | 服务器 WSS `package.key` 中转 | P2P E2E 通道（优先） / .wlk 文件（备用） |
| groupId 来源 | 服务器群组的 `groups.id` | `SHA256(组串)`，组网时各自计算 |
| 公钥查询 | `GET /api/users/{id}/public-key` | 组网时已交换证书，本地获取 |
| 服务端参与 | 密钥中转 + 群成员身份校验 | **零参与** |

### 8.1 JWT 策略

| 配置项 | 值 | 说明 |
|--------|-----|------|
| Access Token 有效期 | 2 小时 | 短时效，减少泄漏风险 |
| Refresh Token 有效期 | 30 天 | 长时效，仅用于续期 Access Token |
| Refresh 轮换 | 每次使用发放新 Refresh Token | 旧 Refresh Token 立即失效 |
| 存储方式 | Access Token 存内存 / Refresh Token 存 HttpOnly Cookie 或安全本地存储 | 不存 localStorage |
| 401 处理 | 客户端自动用 Refresh Token 续期，失败则跳转登录页 | — |
| 吊销 | 服务端维护黑名单（内存），Refresh Token id 加入黑名单即吊销 | 用户改密码/管理员禁用时触发 |
| Token 载荷 | `{sub, username, email, role, iat, exp}` | 不含敏感信息 |

### 8.4 验证码策略

| 配置项 | 值 | 说明 |
|--------|-----|------|
| 形式 | 6 位数字 | 手机/桌面均可阅读 |
| 有效期 | 5 分钟 | 超时需重新发送 |
| 错误次数 | 5 次 | 超过后 code 作废，需重新发送 |
| 重发间隔 | 60 秒 | 同一邮箱 60s 内拒绝重发 |

| 存储 | 内存 Map | 不持久化，进程重启全失效 |

## 九、WebSocket 消息协议

```
客户端 → 服务器:
  sync.ping          {clientTime}                          # 时钟同步
  session.start      {groupId, fileHash, transferMode,     # 发起共享
                       fileName, duration, magnetUri?,
                       lanHost?, lanPort?, ...}
  session.join       {sessionId}                           # 加入共享
  session.leave      {sessionId}
  session.end        {sessionId}                           # 结束共享
  playback.play      {sessionId, position}                 # 播放 (仅host)
  playback.pause     {sessionId, position}                 # 暂停 (仅host)
  playback.seek      {sessionId, position}                 # 跳转 (仅host)
  playback.next      {sessionId, trackHash}                # 切歌 (音乐)
  chat.send          {groupId, content, sessionId?}
  p2p.signal         {sessionId, targetUserId, type, sdp}  # WebRTC信令
  package.key        {groupId, packageHash, key,           # 离线包AES密钥上传
                       expiresIn}                            (打包者→服务器)
  typing.start       {groupId}                             # 开始输入
  typing.stop        {groupId}                             # 停止输入

服务器 → 客户端:
  sync.pong          {clientTime, serverTime}
  session.created    {sessionId, hostId, fileHash, fileName,
                       transferMode, magnetUri?,
                       lanHost?, lanPort?, ...}
  session.ended      {sessionId}
  session.user_joined {sessionId, userId, nickname}
  session.user_left   {sessionId, userId}
  playback.state     {sessionId, position, playing, serverTimestamp}
  chat.message       {groupId, senderId, nickname, content, timestamp, sessionId?}
  p2p.signal         {sessionId, fromUserId, type, sdp}   # WebRTC信令转发
  package.key        {packageHash, exporterId, key,        # 离线包密钥转发
                       expiresAt}                            (服务器→群成员)
  typing.status      {groupId, userId, nickname}           # 某用户正在输入
  error              {code, message}
```

### 9.1 错误码

以下错误码 WebSocket 和 REST API 共用（REST 在 JSON body 的 `error` 字段，WS 在 `error.code` 帧）：

| code | 说明 | 触发场景 |
|------|------|----------|
| `AUTH_EXPIRED` | Token 已过期 | JWT 过期，需重新登录 |
| `AUTH_INVALID` | Token 无效 | 伪造或已吊销的 token |
| `EMAIL_NOT_VERIFIED` | 邮箱未验证 | 登录时 status=PENDING，需先验证邮箱 |
| `EMAIL_TAKEN` | 邮箱已被注册 | 注册时 email 已存在 |
| `USERNAME_TAKEN` | 用户名已被占用 | 注册时 username 已存在 |
| `BAD_CREDENTIALS` | 账号或密码错误 | 登录失败 |
| `CODE_INCORRECT` | 验证码错误 | 邮箱验证/密码重置时 code 不匹配 |
| `CODE_EXPIRED` | 验证码已过期 | 超过 5 分钟 |
| `CODE_EXHAUSTED` | 验证码已作废 | 连续 5 次错误 |
| `RATE_LIMITED` | 请求频率超限 | WS 帧/API 调用超过限流阈值 |
| `SESSION_NOT_FOUND` | 会话不存在 | 加入已结束的会话 |
| `SESSION_FULL` | 会话参与人数已满 | 超过 maxSessionParticipants |
| `PERMISSION_DENIED` | 操作被当前播放模式禁止 | 主持人模式下非 host 发 playback 指令 |
| `GROUP_NOT_MEMBER` | 非群组成员 | 尝试操作未加入的群组 |
| `PACKAGE_KEY_EXPIRED` | 离线包密钥已过期 | 超过 24h TTL 且无 .wlk 私钥文件 |
| `INVALID_MESSAGE` | 消息格式错误 | WS 帧缺少必填字段或类型未知 |
| `SERVER_ERROR` | 服务器内部错误 | 未预期的运行时异常 |

## 十、播放同步机制（不变的核心）

### 10.1 时钟同步
```
Client → sync.ping(t1) → Server 记录收到时间 t2 → Server 返回 {t1, t2}
Client 收到(t3)，计算: offset = t2 - (t1 + (t3 - t1)/2)
每 30 秒采样一次，取最近 5 次的中位数作为当前 offset
```

### 10.2 同步策略
- 偏差 < 100ms：不调整
- 偏差 100~500ms：微调播放速率 0.95x~1.05x
- 偏差 > 500ms：直接 seek 到目标位置

### 10.3 音乐场景独立处理

音乐播放与视频播放使用独立的 UI 页面（`MusicSessionView`），协议层共用同一套 `playback.*` 消息，但增加以下专属逻辑：

- **切歌缓冲窗口**：新曲目开始前 2 秒广播 `playback.next`，所有客户端就位后开始播放
- **歌单队列**：播放列表由服务端 `playlist_tracks` 管理，客户端同步展示
- **歌词同步**：本地加载 .lrc 文件，跟随播放进度滚动

| 手动同步 | 一键「同步到主持人」按钮，立即 seek 对齐进度（所有模式均可用） | ✅ | ✅ |

### 12.2 文件传输方式选择

用户可在会话中选择传输方式。可用的传输方式由服务器 `capabilities` 接口和客户端平台能力共同决定。各传输方式的技术细节见第四节。

### 12.3 上传带宽优化（防卡顿）

发起者上行带宽有限时，其他用户容易卡顿。以下策略从多个层面解决：

| 策略 | 说明 |
|------|------|
| **分段优先级下载** | 优先下载当前播放位置附近的数据块，保证当前播放不卡顿，历史/未来部分后台慢慢下 |
| **多源并行下载** | 同时从多个 peers 下载不同数据段，避免单一来源瓶颈 |
| **上传限速保护** | 发起者可设置上传上限（1/2/5 MB/s），保留带宽给自己播放 |
| **「我下完了，我来当种」** | 客户端下载完整个文件后自动切换为 seeder，替发起者分担上传压力 |
| **慢速预警** | 下载速度 < 播放码率时，UI 提示「下载速度不足，建议暂停缓冲或等待其他 peers」 |
| **预缓冲智能调整** | 检测到下载速度波动大时，自动将预缓冲量从 5s 提升到 15s，减少卡顿 |
| **WiFi 种子节点** | 检测到本机在 WiFi 下且下载完成，自动延长做种时间；移动数据下减少上传 |
| **发起者最低上传保障** | 发起者上传带宽低于文件码率 × 1.5 时，弹窗提醒「你的上行带宽可能不足以支持多人同步观看，建议邀请已下载完成的用户加入作为种子节点」 |

### 12.4 运营商 P2P 限流规避

**上传模式（用户可选）：**

| 模式 | 行为 |
|------|------|
| **纯下载**（默认） | 只下载不上传，下完即停。客户端种子引擎设 upload=0、拒绝传入连接 |

**持续防护：**

| 策略 | 说明 |

## 十四、服务降级保护

2核2G 服务器资源有限，必须在多层设置保护，避免单点异常拖垮整个服务。

### 14.1 限流 (Rate Limiting)

```
┌──────────────────────────────────────────────┐
│                  Nginx 层                     │
│  limit_req_zone: 单IP 30 req/s               │
│  limit_conn: 单IP 最大 10 并发连接             │
├──────────────────────────────────────────────┤
│              Spring Boot 层                   │
│  Resilience4j RateLimiter                     │
│  /api/auth/login   : 5 次/分钟/IP (防暴力破解) │
│  /api/auth/register : 3 次/分钟/IP (防批量注册)   │
│  /api/auth/verify   : 3 次/分钟/邮箱 (防暴力穷举)  │
│  /api/auth/* 发邮件 : 1 次/60秒/邮箱 (防刷SMTP)   │
│  /api/*             : 60 次/分钟/用户             │
│  同 IP 每日发邮件   : 20 封/天                    │
│  WebSocket 消息     : 30 条/秒/连接               │
│  P2P 信令转发       : 10 条/秒/连接               │
├──────────────────────────────────────────────┤
│              应用层保护                        │
│  单文件 hash 校验 : 限制 100MB，超限拒绝        │
│  WebSocket 消息体 : 最大 64KB                  │
│  session 参与者上限: 20 人/会话                │
│  单用户并发会话    : 5 个                       │
└──────────────────────────────────────────────┘
```

### 14.2 熔断 (Circuit Breaker)

| 保护目标 | 策略 | 参数 |
|----------|------|------|
| 数据库查询 | 慢查询熔断 | > 3s 触发，30s 后半开 |
| WebSocket 广播 | 线程池隔离 | 专用线程池 4 线程，队列 100 |
| P2P 信令转发 | 熔断 + 降级 | 失败率 > 50% → 停止转发，告知客户端直连 |
| 聊天消息批量写入 | 限时重试 | 重试 3 次，间隔 200ms，全失败 → 仅广播不持久化 |

### 14.3 线程池隔离

```
主业务线程池 (Tomcat)      WebSocket 线程池         后台任务线程池
  core: 10, max: 20         core: 4, max: 8           core: 2, max: 4
  queue: 100                queue: 50                 queue: 20
      │                         │                         │
  REST API 请求             播放同步 / 聊天 / 信令      缩略图生成(可选)
                             消息分发                   错误日志批量写入
```

超过队列容量的请求直接返回 `503 Service Unavailable`，不排队等待。

### 14.4 降级策略与错误记录

| 场景 | 降级行为 | 错误日志记录 |
|------|----------|-------------|
| 数据库连接超时 | 返回缓存数据（若有），否则 503 | ERROR + 写入 error_logs 表 |
| WebSocket 广播失败 | 仅发给在线用户，失败的不重试 | WARN |
| P2P 信令转发失败 | 返回对端公网 IP，客户端尝试直连 | WARN + 标记降级处理 |
| 内存使用 > 80% | 拒绝新会话创建，返回 "服务器繁忙" | ERROR |
| 单用户消息频率超限 | WebSocket 帧丢弃，返回 rate_limit 错误 | INFO |
| 恶意文件 hash (超大) | 直接拒绝，不通知群组 | WARN |

### 14.5 降级触发时的应对措施

每种保护触发后，系统需要给客户端**明确的反馈**，同时做好**自动恢复**：

| 触发场景 | 客户端看到什么 | HTTP/WS 状态 | 系统动作 | 恢复条件 |
|----------|---------------|-------------|----------|----------|
| 登录限流 (5次/分) | "操作太频繁，请1分钟后再试" | 429 | 记录 IP，连续3次429后封IP 15分钟 | 15分钟后自动解封 |
| API 限流 (60次/分) | "请求过于频繁，请稍后重试" | 429 + Retry-After 头 | 计数器累加，超阈值告警 | 下一分钟窗口自动重置 |
| WebSocket 消息超限 | `{type:"error", code:"RATE_LIMITED", retryAfter:3}` | WS 帧 | 该连接后续 3 秒消息丢弃 | 3 秒后恢复 |
| 数据库慢查询熔断 | "服务暂时不可用，请稍后重试" | 503 | 熔断器打开，所有 DB 请求快速失败 | 30s 后半开，连续 3 次成功则关闭 |
| 线程池队列满 | "服务器繁忙，请稍后重试" | 503 | 直接拒绝，不排队；WARN 日志 | 队列降至 50% 以下 |
| WebSocket 广播线程池满 | 部分用户可能未收到消息，下次心跳自动同步 | 静默丢弃非关键帧 | 丢弃的消息不重试，客户端下次 sync.ping 时补发状态 | 队列恢复 |
| P2P 信令熔断 | 返回对端外网 IP，客户端尝试直连 (ICE) | 正常 200 + extra 字段含 `fallback: "direct"` | 信令中继停止，客户端走 STUN/TURN 直连 | 30s 后半开 |
| 内存 > 80% | "服务器繁忙，暂时无法创建新会话" | 503 | 拒绝 session.start；现有会话不受影响 | GC 后 < 70% |
| 内存 > 90% | 所有请求返回 503 + 主动断掉空闲 WS 连接 | 503 + WS close | 逐出空闲连接（5分钟无活动），保护 JVM 不 OOM | GC 后 < 70% |
| 单文件 > 100MB | "文件过大（最大100MB），请压缩后重试" | 400 | 拒绝哈希注册，不广播，不存元数据 | 用户换小文件 |
| 恶意快速建群 | "操作过于频繁" | 429 | 冻结该用户建群权限 30 分钟 | 30 分钟后自动解冻 |

### 14.6 客户端应对策略

客户端也需要配合降级，不能无脑重试：

```
// 客户端收到 429 或 rate_limit 错误时
收到 429:
  1. 读取 Retry-After 响应头
  2. 进入退避，在 Retry-After 时间内不重试
  3. 连续 3 次 429 → 退避指数增长 (最长 5 分钟)

收到 503:
  1. 随机等待 2~5 秒后重试
  2. 连续 3 次 503 → 提示用户 "服务器繁忙，请稍后刷新页面"
  3. 每 30 秒发一次探测请求，成功后恢复正常

WebSocket 突然断开:
  1. 立即重连一次
  2. 失败则退避: 2s → 5s → 10s → 30s (上限)
  3. 断线期间播放器继续本地播放，不中断
```

### 14.7 优雅关闭

```
收到 SIGTERM
  → Nginx 停止转发新请求 (health check 标记 unhealthy)
  → 等待 30s，处理完进行中的 HTTP 请求
  → 关闭 WebSocket 连接（发送 session.ended 通知）
  → 批量写入未持久化的聊天消息
  → JPA flush → 正常退出
```

### 14.8 监控告警阈值

| 指标 | 触发阈值 | 动作 |
|------|----------|------|
| JVM 堆使用率 | > 85% | 记录 ERROR 日志，拒绝新建会话 |
| WebSocket 活跃连接 | > 800 | WARN 日志，通知管理员 |
| 限流触发次数 | > 100/分钟 | WARN 日志，检查是否有攻击 |
| 熔断器打开 | 任意 | ERROR + 写入 error_logs 表 |
| 磁盘剩余 | < 5GB | WARN 日志 |

## 十五、基础安全防护

3Mbps 小带宽，在无外部 CDN/高防的情况下，能做的是把服务器自身守好。

| 层级 | 措施 | 效果 |
|------|------|------|
| 服务商安全组 | 仅开放 80/443/22（密钥登录） | 最小攻击面 |
| iptables | SYN cookie + 连接速率限制 | 防小规模 SYN flood |
| Nginx | `client_max_body_size 1m` + `client_body_timeout 10s` | 防大包 + 慢速攻击 |
| Nginx | `limit_req_zone` 单 IP 30 req/s | 防单 IP 高频请求 |
| Nginx | `limit_conn` 单 IP 20 并发 | 防连接耗尽 |
| Spring Boot | 仅监听 127.0.0.1 | 不对外暴露端口 |
| fail2ban | 3 次失败封 1 小时（SSH + Nginx 404 扫描） | 自动封扫描 IP |
| 带宽上限 | 超过 3Mbps → 服务自然降级 | 物理上限本身就是防护 |

超出 3Mbps 的攻击只能靠外部清洗，暂不做。先把这层基础防护做到位。

| 风险 | 对策 |
|------|------|
| WebTorrent 在移动浏览器兼容性差 | Android 用原生 App，Web 端提示用 Chrome/Edge |
| 客户端 NAT 严格导致 P2P 不通 | 服务器部署 STUN/TURN（coturn），TURN 中继仅作兜底 |
| 用户本地无文件、P2P 下载慢 | 5 种传输方式互为备选，WebTorrent 边下边播 |
| 文件 hash 碰撞 | 用 SHA256（碰撞概率可忽略）+ 文件大小双重校验 |
| 安卓 BitTorrent 库选型 | 优先测试 libtorrent4j，备选 FrostWire |
| P2P 下载占满发起者上行带宽 | 客户端限制上传速率（默认 2MB/s），可配置 |
| LAN 传输中 WiFi 断开切移动数据 | 传输期间锁定 WiFi（Android ConnectivityManager），切网自动暂停；流量保护模式下拒绝移动数据传输 |
| 离线包密钥 24h 过期导致无法解包 | 导入时客户端自动检测密钥状态，过期则提示「密钥已过期，请联系打包者重新分享」；打包者可一键补发密钥

## 十八、服务器部署

```
/opt/wltogether/
├── app.jar
├── config/application.yml
├── data/
│   ├── wl.db              # SQLite 数据库文件
│   └── avatars/           # 用户/群组头像 (WebP)
├── logs/
└── web-dist/              # Vue 构建产物

# 如果使用 MySQL，数据库文件在 MySQL 数据目录，此处无需 data/
```

注意：没有 `media-storage/` 目录——服务器不存任何媒体文件。

## 十九、验证计划

1. **单元测试**: Service 层逻辑
2. **集成测试**: WebSocket 信令中继 + 播放同步指令流
3. **端到端测试**:
   - 注册两个用户 → 建群组 → A 发起会话(选本地文件) → B 加入
   - B 通过 WebTorrent 从 A 下载文件 → 边下边播
   - A 点击播放 → 验证 B 同步播放，偏差 < 500ms
   - A 拖动进度条 → 验证 B 自动 seek
   - 聊天消息实时收发
   - B 本地已有相同 hash 文件 → 跳过下载直接播放
